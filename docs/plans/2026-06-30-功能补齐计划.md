## 第一阶段：让系统活起来（数据管线 + 认证）

### Task 1-1: JWT 认证系统

**为什么优先:** 没有认证，整个系统无法投入实际使用。当前所有 API 都是裸奔状态。

**后端:**
- 创建 `api/dependencies.py` — JWT bearer token 依赖注入, `get_current_user(role)`
- 创建 `api/routers/auth.py` — `POST /api/auth/login`, `POST /api/auth/register`, `POST /api/auth/refresh`
- 创建 `api/models/user.py` — User ORM 模型 (id, username, password_hash, role, is_active)
- DDL: users 表 (如果不存在)
- 给现有路由加上 `Depends(get_current_user)` 保护

**前端:**
- 创建 `frontend/src/app/login/page.tsx` — 登录页面
- 创建 `frontend/src/lib/auth.ts` — token 存储/刷新/拦截器
- `api.ts` 添加 `Authorization: Bearer` 头

**文件清单:**
- NEW: `api/dependencies.py`
- NEW: `api/routers/auth.py`
- NEW: `api/models/user.py`
- NEW: `frontend/src/app/login/page.tsx`
- NEW: `frontend/src/lib/auth.ts`
- MODIFY: `api/main.py` (注册 auth router)
- MODIFY: `frontend/src/lib/api.ts` (加 auth header)
- MODIFY: 所有 api/routers/*.py (加 Depends 保护)

## 不做的事

以下功能有后端代码但没有前端入口，且业务价值不明确，**暂不纳入计划**：

- 本体引擎管理界面 (ontology 编辑)——目前 config/ontology.json 手动维护即可
- 冲突解决界面 (conflict_resolver)——使用频率极低
- 字段历史查看器 (field_history)——可以后续在来源追溯里顺带做
- 术语纠正管理界面 (term_corrector)——已有自动化逻辑，无需手动管理
- 数据质量报告页面——可在数据中枢里加一个 tab，但不作为独立页面
