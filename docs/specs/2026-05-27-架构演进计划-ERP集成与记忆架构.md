### 1.4 已识别的架构问题

1. **工单数据是死的**：plan_end全空，无法计算交期风险
2. **缺料靠手动**：ERP有自动缺料报告，本地却用手动录入
3. **没有BOM**：无法拆解工单为工序，排产无从谈起
4. **没有版本概念**：INSERT OR REPLACE覆盖旧数据，无法追溯变更
5. **标准工时和工单脱节**：std_time_units是装配单元基准，work_orders没有关联

## 三、目标架构

### 3.1 三层数据模型

```
┌─────────────────────────────────────────────────────┐
│                   本地 SQLite (factory.db)            │
│                                                      │
│  ┌── ERP缓存层（IN3同步，可重新拉取）──────────────┐  │
│  │  work_orders    ← MO API（工单+交期+状态）      │  │
│  │  material_bom   ← BOM API（母件→子件展开）      │  │
│  │  stock_snapshots← Stock API（库存快照）         │  │
│  │  shortages      ← Shortage API（缺料报告）      │  │
│  └──────────────────────────────────────────────────┘  │
│                                                      │
│  ┌── 现场数据层（人工录入，ERP没有）──────────────┐  │
│  │  daily_reports  每日报工                       │  │
│  │  leave_records  请假记录                       │  │
│  │  skill_matrix   技能矩阵                       │  │
│  │  employees      员工配置                       │  │
│  └──────────────────────────────────────────────────┘  │
│                                                      │
│  ┌── 记忆层（系统自动写入+人工补充）──────────────┐  │
│  │  memory_versions   版本变更记录                 │  │
│  │  memory_snapshots  状态快照（每日）             │  │
│  │  memory_events     异常事件+决策审计            │  │
│  └──────────────────────────────────────────────────┘  │
│                                                      │
│  ┌── 配置层───────────────────────────────────────┐  │
│  │  config          系统参数                       │  │
│  │  std_time_units  装配单元基准                   │  │
│  └──────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

### 3.2 数据流向

```
IN3 ERP (云)                      本地系统 (SQLite)              页面 (Streamlit)
    │                                    │                            │
    │── MO工单 ──→ erp_sync.py ──→ work_orders ──→ 工单进度/交期风险 ──→ 页面
    │── BOM ────→ erp_sync.py ──→ material_bom ──→ (排产引擎) ────→ 页面
    │── 缺料 ───→ erp_sync.py ──→ shortages ───→ 缺料跟踪 ──────→ 页面
    │── 库存 ───→ erp_sync.py ──→ stock_snapshots → (排产参考) ───→ 页面
    │                                    │                            │
    │                                    │←── save_daily_report() ←── 报工页面
    │                                    │←── save_leave_record() ←── 请假页面
    │                                    │←── memory_versions ←──── 标准工时修正
    │                                    │←── memory_snapshots ←─── 定时任务
    │                                    │←── memory_events ←────── 异常检测
```

### 3.3 记忆架构设计

参照SAP的 Knowledge Graph → Memory → Governance 三层模型：

| 层级 | 对应 | 我们的实现 |
|------|------|-----------|
| Knowledge Graph | 实体关系 | work_orders↔material_bom↔employees↔std_time_units |
| Memory | 时间上下文 | memory_versions（变更链）+ memory_snapshots（时间序列） |
| Governance | 规则+审计 | memory_events（事件+决策+措施）+ config（规则参数） |

**每个场景的学习闭环**：

```
标准工时：基准值 → 报工偏差 → 原因标注 → 修正基准 → 版本记录
交期风险：今日风险 → 每日快照 → 趋势对比 → 预警升级
加班管控：本月加班 → 月度快照 → 模式识别 → 提前预警
缺料跟踪：ERP缺料 → 每日同步 → 趋势追踪 → 关联交期风险
```

## 四、新增数据库Schema

### 4.1 ERP缓存表

```sql
-- BOM展开（母件→子件）
CREATE TABLE IF NOT EXISTS material_bom (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    parent_material_no TEXT NOT NULL,        -- 母件编码
    parent_material_id TEXT,                 -- 母件UUID
    child_material_no TEXT NOT NULL,         -- 子件编码
    child_material_id TEXT,                  -- 子件UUID
    quantity REAL NOT NULL DEFAULT 0,        -- 用量（多分组累加）
    synced_at TEXT NOT NULL DEFAULT (datetime('now','localtime')),
    UNIQUE(parent_material_no, child_material_no)
);
CREATE INDEX IF NOT EXISTS idx_bom_parent ON material_bom(parent_material_no);
CREATE INDEX IF NOT EXISTS idx_bom_child ON material_bom(child_material_no);

-- 库存快照（每次同步写一条）
CREATE TABLE IF NOT EXISTS stock_snapshots (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    material_no TEXT NOT NULL,
    material_id TEXT,
    material_name TEXT,
    on_hand_qty REAL DEFAULT 0,
    freeze_qty REAL DEFAULT 0,
    po_qty REAL DEFAULT 0,
    arrival_qty REAL DEFAULT 0,
    pr_qty REAL DEFAULT 0,
    mo_pending_qty REAL DEFAULT 0,
    mo_producting_qty REAL DEFAULT 0,
    available_qty REAL DEFAULT 0,
    snapshot_date TEXT NOT NULL,              -- YYYY-MM-DD
    synced_at TEXT NOT NULL DEFAULT (datetime('now','localtime'))
);
CREATE INDEX IF NOT EXISTS idx_ss_material ON stock_snapshots(material_no);
CREATE INDEX IF NOT EXISTS idx_ss_date ON stock_snapshots(snapshot_date);
```

### 4.4 记忆表

```sql
-- 版本变更记录
CREATE TABLE IF NOT EXISTS memory_versions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    entity_type TEXT NOT NULL,      -- 'std_time_unit' / 'work_order' / 'employee' / 'config'
    entity_id TEXT NOT NULL,        -- 对应主键值
    field_name TEXT NOT NULL,       -- 变更字段
    old_value TEXT,
    new_value TEXT,
    reason TEXT,                    -- 变更原因
    changed_by TEXT,                -- 操作人
    created_at TEXT NOT NULL DEFAULT (datetime('now','localtime'))
);
CREATE INDEX IF NOT EXISTS idx_mv_entity ON memory_versions(entity_type, entity_id);
CREATE INDEX IF NOT EXISTS idx_mv_date ON memory_versions(created_at);

-- 状态快照（每日定时写入）
CREATE TABLE IF NOT EXISTS memory_snapshots (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    snapshot_type TEXT NOT NULL,    -- 'delivery_risk' / 'overtime' / 'efficiency' / 'shortage'
    entity_id TEXT NOT NULL,        -- order_id 或 employee
    metric_name TEXT NOT NULL,      -- 'risk_level' / 'overtime_hours' / 'efficiency_rate'
    metric_value TEXT NOT NULL,
    context TEXT,                   -- JSON附加信息
    snapshot_date TEXT NOT NULL,
    created_at TEXT NOT NULL DEFAULT (datetime('now','localtime'))
);
CREATE INDEX IF NOT EXISTS idx_ms_type ON memory_snapshots(snapshot_type, entity_id);
CREATE INDEX IF NOT EXISTS idx_ms_date ON memory_snapshots(snapshot_date);

-- 异常事件+决策审计
CREATE TABLE IF NOT EXISTS memory_events (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    event_type TEXT NOT NULL,       -- 'anomaly' / 'warning' / 'decision' / 'pattern'
    source TEXT NOT NULL,           -- 'report' / 'overtime' / 'delivery' / 'material' / 'erp_sync'
    entity_id TEXT,                 -- 关联实体
    severity TEXT DEFAULT 'info',   -- 'info' / 'warning' / 'critical'
    title TEXT NOT NULL,
    detail TEXT,
    action_taken TEXT,              -- 人工干预措施
    resolved INTEGER DEFAULT 0,
    created_at TEXT NOT NULL DEFAULT (datetime('now','localtime'))
);
CREATE INDEX IF NOT EXISTS idx_me_type ON memory_events(event_type, source);
CREATE INDEX IF NOT EXISTS idx_me_date ON memory_events(created_at);
```

## 六、关键设计决策

| # | 决策 | 原因 |
|---|------|------|
| 1 | **本地SQLite保留，不换PostgreSQL** | 数据量在万级，SQLite足够；单文件部署简单 |
| 2 | **ERP数据是缓存，不是源** | 本地可随时重新同步，ERP才是权威源 |
| 3 | **现场数据（日报/请假/技能）只在本地** | ERP不提供，这是我们的独有价值 |
| 4 | **记忆表独立，不混入业务表** | 版本/快照/事件是纵向切片，跨所有实体 |
| 5 | **复用right老师的auth.py，不重写** | 鉴权逻辑已经验证过，直接复制 |
| 6 | **同步脚本独立于Streamlit** | 可命令行运行，也可在页面触发 |
| 7 | **不存储工资金额** | 冯总要求：只存薪资等级（大工/中工/普工），不存具体金额 |
| 8 | **admin密码门禁保持** | 简单够用，将来需要时再升级 |

## 八、风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| IN3 API变更/不可用 | 工单数据过期 | 同步失败时保留上次数据+memory_events记录 |
| Token过期 | 同步中断 | auth.py已有自动刷新逻辑 |
| ERP数据和日报数据冲突 | 工单号不匹配 | 以ERP工单号为准，日报中的order_id做模糊匹配 |
| 同步频率过高 | API限流 | 默认启动时同步一次，手动可触发 |
| BOM层级过深 | 展开复杂 | 暂时只展开一层，够排产用 |
