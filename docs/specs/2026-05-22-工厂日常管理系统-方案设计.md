## 核心结论：产品定位升级

厂长原话：

> "不是单纯的工时管理了，应该叫类似工厂日常管理。虽然我要的是工时，但你给我做的还是工厂的东西。"

```
工时管理看板 → 工厂日常管理驾驶舱
工时是切入点，管理是终点。
```

| 功能 | 说明 | 状态 |
|------|------|------|

| 标准工时对比 | 子族Q1 × 关键词系数 × 学习宽放 | ✅ 已运行 |

| 缺料记录 | 手动登记缺料项、采购承诺跟踪 | ⚠️ 部分覆盖 |
| 数据可信度标注 | A/B/C/D 四级置信度 | ✅ 本周新增 |

### 第二层：管得住（下一步）

> 管理闭环——发现异常 → 预警 → 干预。

| 功能 | 厂长原话 | 关键设计点 |
|------|---------|-----------|
| **加班管控** | "劳动法要求控制在36小时。我可以调比例，改成48他就提醒我这个人已经超了" | 可配置阈值、按薪资分级控制、超标标红 |
| **交货期风险色标** | "快交付了，黄色提醒；距离还差一周，绿色；超了就变红、甚至变紫变黑" | 连ERP交期、自动算剩余天数、绿黄红紫四级 |
| **病事假管理** | "病假关注健康度，事假关注这人有什么问题" | 病假→健康预警、事假→出勤异常标记 |
| **缺料自动关联** | "连ERP系统，直接显示缺料，能不能干活" | 工单→BOM→库存→自动标缺料状态 |

### 第三层：能决策（远期）

> 智能辅助——系统建议，人做决策。

| 功能 | 厂长原话 | 关键设计点 |
|------|---------|-----------|
| **技能矩阵** | "人员覆盖的设备，钳工还是电工，具体到单头铣。一人掌握两个技能，我不让他覆盖所有，对咱企业是风险" | 基础技能+设备级标签、效率评分、风险控制规则 |

| **语音交互查询** | "我要什么就跟我说，类似小慧。我一跟他说，他能调出来我要的东西" | ASR→NLP→数据查询→可视化反馈 |
| **预测性排产** | "新工单来了，预估工时能不能按期交付" | 标准工时+学习曲线+可用产能+交期 |

## 二、各功能模块详细设计

### 2.1 加班管控

**场景**：管理者设定月度加班上限，系统自动监控每人累计加班工时，超标预警。

**核心逻辑**：
```
月度加班工时 = Σ(每日工时 - 标准日工时)
预警阈值：默认36h/月（可配置）
超标判定：累计加班 ≥ 阈值 → 标红
```

**分级控制**（厂长要求）：
| 薪资等级 | 默认阈值 | 管控逻辑 |
|---------|---------|---------|
| 高薪（大工） | 36h | 严格控制，超标即时预警 |
| 中薪 | 48h | 中等控制 |
| 普薪 | 60h | 宽松控制 |

**界面表现**：
- 人员视图中新增"本月加班"列
- 超阈值行标红底色
- 阈值在后台可配置（隐藏面板，权限控制）

**数据需求**：
- 标准日工时（默认10.5h）
- 月度加班阈值配置
- 人员薪资等级（用于分级）

---

### 2.2 交货期风险预警

**场景**：工单列表自动标注距交货期天数，按风险等级着色。

**核心逻辑**：
```
剩余天数 = 计划完工日 - 今天
风险等级：
  已超期（剩余 < 0）        → 紫色/黑色（严重）
  紧急（0 ≤ 剩余 ≤ 3天）    → 红色
  警告（4 ≤ 剩余 ≤ 7天）    → 黄色
  正常（剩余 > 7天）        → 绿色
  无交期数据                → 灰色
```

**结合工时进度**：
```
预计还需工时 = 标准工时 - 已耗实际工时
预计还需天数 = 预计还需工时 / 近7天日均工时
交付风险 = 预计还需天数 > 剩余天数 → "可能延期"
```

**界面表现**：
- 工单视图中新增"交期"列和"风险"色标列
- 点击可展开：计划完工日、剩余天数、预计还需工时、风险预判

**数据需求**：
- ERP 工单计划完工日（当前ERP导出有此字段）
- 标准工时（已有）
- 实际工时（已有）

---

### 2.3 病事假管理

**场景**：记录每人病假/事假天数，联动出勤统计和健康预警。

**核心逻辑**：
```
出勤天数 = 应出勤天数 - 病假 - 事假 - 其他假
```

**健康度预警（病假）**：
| 周期 | 阈值 | 预警 |
|------|------|------|
| 月度 | 病假 ≥ 3天 | 🟡 关注 |
| 月度 | 病假 ≥ 5天 | 🔴 健康预警 |
| 季度 | 病假 ≥ 8天 | 🔴 严重关注 |

**出勤异常（事假）**：
| 模式 | 预警 |
|------|------|
| 连续2天事假 | 🟡 关注 |
| 月度事假 ≥ 5天 | 🔴 异常标记 |
| 周五/周一频繁事假 | 🟡 模式标记 |

**数据需求**：
- 每日请假记录（类型+天数）
- 应出勤日历

---

### 2.4 技能矩阵

**场景**：建立人员-技能-设备的映射关系，支撑派工决策和风险管控。

**技能定义层级**：
```
Level 1 — 工种：钳工 / 电工 / 调试
Level 2 — 设备族：族A单机加工 / 族B龙门仓储 / 族C生产线
Level 3 — 具体设备：冲剪机 / 铜排仓 / 单头铣 / 双头铣 / 折弯机 / 抽屉线 / 打标机
Level 4 — 效率评级：该设备上该人员的效率等级（快/中/慢）
```

**风险控制规则**（厂长明确要求）：
```
每人核心技能 ≤ 2 项设备
原因：降低单点依赖——一个人走了，不能所有设备都停
```

**界面表现**：
- 人员视图中新增"技能标签"列（简洁模式）
- 独立"技能矩阵"视图：人员×设备的网格图，颜色表示效率
- 风险视图：每项设备有几个掌握者？少于2人标红

**数据需求**：
- 人员-设备技能映射（IE评估录入，初始一次性建立，后续维护）
- 效率评级可从事后工时数据自动推算

### 2.5 派工与人员组合优化

**场景**：新工单到来时，系统根据技能匹配+产能+交期，给出派工建议。

**输入**：
- 待派工单（工单号、设备类型、技能要求、预估工时、交期）
- 可用人员（在岗、技能匹配、剩余产能）
- 约束条件（加班上限、技能风险控制）

**输出**：
```
派工建议：
  工单XXX（铜排仓，预估200h，交期6/15）
  → 主责：张三（铜排仓-快，剩余产能80h）
  → 辅助：李四（铜排仓-中，剩余产能120h）
  → 风险：需5/28前启动，否则交期风险↑
```

**核心算法思路**（非本期实现，先记录方向）：
```
1. 匹配：工单技能要求 ∩ 可用人员技能
2. 排序：按效率评级 + 剩余产能
3. 分配：主责优先满配，辅助补充
4. 校验：技能风险（每设备≥2人会） + 加班上限 + 交期可行性
5. 输出：派工建议单 + 风险标记
```

**数据需求**：
- 技能矩阵（前置依赖）
- ERP工单+交期（前置依赖）
- 加班阈值配置（前置依赖）

### 2.6 语音交互查询

**场景**：管理者通过语音提问，系统理解意图后返回结果。

**典型对话示例**：
```
用户："这周效率最高的三个工单是哪些？"
系统：展示效率排行Top3，语音播报摘要

用户："谁这个月加班超过36小时了？"
系统：列表展示超标人员

用户："521工单什么时候交货？还差多少工时？"
系统：展示交期+进度+风险预判

用户："铜排仓现在谁在做？每个人花了多少？"
系统：按人员列出该设备族的工时分布
```

**技术思路**（远期，先记录方向）：
```
语音输入 → ASR（语音识别） → NLP意图解析 → 数据查询 → 可视化 + 语音播报
```

**意图分类**（预定义查询模板）：
| 意图 | 触发词 | 查询逻辑 |
|------|--------|---------|
| 工单效率排行 | "效率最高/最低" | Top N by efficiency |
| 人员工时查询 | "谁/哪个人/XX干了多少" | Filter by employee |
| 加班预警 | "加班超过/超了" | Filter by overtime > threshold |
| 交货风险 | "什么时候交/还差多少/能不能交" | Filter by order + risk calc |
| 设备工时 | "XX设备/铜排仓/冲剪机" | Filter by product family |

### 2.7 缺料自动关联

**场景**：从ERP/MES获取物料状态，自动标注工单是否因缺料停工。

**当前状态**：手动记录5个工单的缺料情况，不完整。

**目标流程**：
```
ERP工单 → BOM展开 → 查物料库存/采购状态
  ↓

  ↓
工单视图中显示缺料状态标签
```

**数据需求**：
- ERP物料库存数据
- ERP采购订单+在途状态
- BOM数据（已有部分）

## 三、工时统计方式演进

### Phase 1 — 当前：纸质→录音→转录

```
员工下班手写纸质记录表
  → 班组长审核、誊写另一份纸质汇总
  → IE通过录音转文字获取数据
  → 手动转录为CSV
  → Python脚本分析 → HTML报告
```

**痛点**：3人经手、链路长、滞后1-2天、易出错。

### Phase 2 — 近期：电子报工 + 自动转录

```
班组长/员工填写电子报工表
  → 语音自动转录（demo已有）
  → 数据直接入库
  → T+0 出当日报告
```

**目标**：消除纸质环节，时效从T+2提升至T+0。

### Phase 3 — 远期：派工制度

```
ERP取生产计划
  → 按工位/技能预分配任务
  → 生成派工单（每人每天做什么）
  → 工人按派工单执行
  → 实际工时回填 → 与计划对比
  → 偏差反馈 → 优化下次派工
```

**目标**：从"事后记录"升级为"事前计划 + 事中跟踪 + 事后分析"。

## 四、标准工时体系

### 4.1 当前方法

```
标准工时 = 子族Q1(P25基准) × 关键词难度系数 × 学习宽放
```

| 因子 | 说明 |
|------|------|
| 子族Q1 | 同产品族已完成工单的下四分位数，取保守值 |
| 关键词系数 | 总装×1.3 / 改制×0.35 / 电气×0.65 / 传输线×0.75 / 联动×1.15 |
| 学习宽放 | 首次×1.30 / 第2次×1.20 / 第3-4次×1.10 / 成熟(≥5次)×1.00 |

### 4.2 置信度升级路径

```
D（默认值）→ C（子族≥2个完工）→ B（子族≥5个+CV<50%）→ A（子族≥10个+CV<30%）
```

每完成一个工单，自动重新计算子族基准，置信度动态更新。

### 4.3 预测新工单工时

```
新工单 → 匹配子族 → 查Q1基准 → 算关键词系数 → 判学习阶段
  → 输出：标准工时 [std, std×1.3] 区间
  → 计划排产使用保守值(std)，风险缓冲用上限(std×1.3)
```

### 4.4 反馈闭环

```
实际工时回写 → 重新计算子族Q1 → 下次预测更准
```

## 六、产品设计原则（来自厂长沟通）

1. **"特别简易，外行一眼看到"** — UI必须是驾驶舱风格，核心指标一眼看完，不要密密麻麻的表格
2. **"类似智能软件，我问你要什么能给我"** — 搜索/过滤/语音是核心交互，不是可有可无
3. **"权限不会给你，但将来要有这个功能"** — 敏感配置项（薪资、阈值）做隐藏面板，权限分级
4. **"我做为管理者的角度去弄"** — 每个功能都要问：管理者看到这个能做什么决策？
5. **"已经做出来的东西就是好事"** — 渐进交付，先有后好
6. **"控制"而非"记录"** — 不只是展示数据，要能设定规则、触发预警
