## 一、项目定位

### 1.1 核心价值主张

这是一个为 **ETO（Engineer-to-Order）铜牌仓设备工厂** 量身打造的 **AI 驱动业务操作系统**（Factory OS）。它不是一个通用的 ERP 或 MES，而是一个深度理解工厂语义的决策中枢。

### 1.2 与通用工具的差异

| 维度 | 通用 BI（帆软/Tableau） | 通用 AI Chat（ChatGPT/Claude） | Factory OS（本项目） |
|------|------------------------|-------------------------------|---------------------|
| **数据理解** | 需要人工定义维度和指标 | 不理解你的业务 | 内置 ontology.json，知道工单/物料/员工的语义关系 |
| **交互方式** | 拖拽报表 | 自然语言对话 | 自然语言 + 结构化决策卡片 + IM 推送 |
| **输出** | 图表 | 文本回复 | 可执行决策（审批 → 执行 → 追踪） |
| **知识沉淀** | 无 | 无 | field_history 全链路数据血缘 |
| **使用者** | 数据分析师 | 任何人 | 工厂班组长 / IE 工程师 / 生产主管 |

### 1.3 一句话

> 让工厂主管用自然语言问"WO-001 为什么延期"，系统不仅告诉原因，还能给出决策建议、等你审批、审批后自动执行、记录所有操作的血缘。

## 二、业务领域

### 2.1 业务对象（Ontology）

系统内置 8 个核心业务对象，定义在 [config/ontology.json](../config/ontology.json)：

```
Worker ──执行──→ WorkOrder ←──生产── Equipment
   │                 │
   │              需要
   │                 │
   ▼                 ▼
AttendanceRecord   Material
                      │
                   库存
                      │
                      ▼
                StockSnapshot
```

每个对象不仅有静态属性，还有**推导属性**（derived properties），由系统自动计算：

| 推导属性 | 计算逻辑 | 所属对象 |
|----------|---------|---------|
| `Worker.monthly_hours` | SUM(daily_reports.hours) 按月聚合 | Worker |
| `WorkOrder.actual_hours` | SUM(daily_reports.hours) 按工单聚合 | WorkOrder |
| `WorkOrder.delay_days` | MAX(0, today - due_date) | WorkOrder |
| `WorkOrder.bom_fulfilled` | 所有 BOM 子件库存 ≥ 需求量 | WorkOrder |
| `Material.required_qty` | SUM(BOM qty × 工单数量) 按物料+生产中工单聚合 | Material |
| `Material.shortage` | required_qty - stock_qty | Material |
| `Equipment.utilization` | actual_run_hours / available_hours × 100 | Equipment |

### 2.2 数据质量标准

标准工时数据按来源分 5 个质量等级：

| 等级 | 标签 | 来源 | 可信度 |
|------|------|------|--------|
| A | 实测 | IE 现场秒表测时 | 高 |
| B | 推导 | 从历史数据统计推算 | 中-高 |
| C | IE 估计 | 工程师经验估算 | 中 |
| D | 默认值 | 系统预设 | 低 |
| E | 未验证 | 无任何依据 | 极低 |

## 五、AI 子系统

### 5.1 能力矩阵

```
                    读                  写
                 ┌──────┐          ┌──────┐
      简单查询   │  ✓   │  写入日报│  ✓   │

      异常检测   │  ✓   │  删除记录│  ✓*  │
      根因分析   │  ✓   │  发消息  │  ✓   │
                 └──────┘          └──────┘
```

### 5.2 架构演进

**Phase 1-5（当前）：语义工具模式**
```
用户输入 → IntentRouter → HubController/ReActAgent
                              ↓
                    40+ 预定义语义工具
                   （每个只做一件事）
```
- 优点：准确、可控
- 缺点：每新增一个分析需求就要加一个工具

**Phase 7（设计中）：原语义工具模式**
```
用户输入 → IntentRouter → ActionDispatcher
                              ↓
                       快捷方式表匹配？
                       /            \
                     是              否
                      ↓              ↓
                 直接执行      ReAct Agent 推理
                              ↓
                       8 个基础原语组合
                       → RiskEngine 分级
                       → 确认/审批
                       → ActionExecutor 执行
```
- 优点：灵活、可组合、LLM 自主推理
- 缺点：LLM 可能犯错，需要风险分级兜底

### 5.3 两个阶段的共存关系

语义快捷方式（3-5 个高频组合）保留了 Phase 1-5 的"确定性准确"优势。新需求走原语推理，成熟的高频需求沉淀为快捷方式。这是一个**渐进式智能化**的路径。

### 6.2 field_history：数据血缘的基石

```sql
CREATE TABLE field_history (
    id INTEGER PRIMARY KEY,
    table_name TEXT,      -- 被修改的表
    record_id TEXT,       -- 被修改的记录 ID
    field_name TEXT,      -- 被修改的字段
    value_old TEXT,       -- 旧值
    value_new TEXT,       -- 新值
    source TEXT,          -- 修改来源 (erp_sync / agent / manual / im)
    recorded_at TEXT,     -- 修改时间
    entered_by TEXT,      -- 操作人
    confidence REAL       -- 数据置信度
);
```

任何写操作都会在 `field_history` 留下一条记录。这意味着**每一条数据都可以追溯**：谁在什么时候、通过什么渠道、以多高的置信度、把什么字段从什么值改成什么值。

## 七、前端架构

### 7.1 页面结构

```
AppShell (固定侧边栏 + 顶栏)
├── 指挥中心 (/)              ← 首页，决策卡片 + KPI
```

### 7.2 设计语言：工业精炼风格 (Industrial Refinement)

- **背景色**：`oklch(0.145 0.012 250)` —— 深碳黑
- **强调色**：蓝色系，工业感
- **字体**：JetBrains Mono（标签/数据），系统字体（正文）
- **设计原则**：克制、精确、可信、高效
- **无装饰元素**：所有视觉元素都有信息含义

## 九、安全模型

### 9.1 认证

- JWT Token 认证（`python-jose`）
- 用户角色：`admin` / `supervisor` / `operator`
- 前端路由按角色过滤菜单可见性
- API 按角色过滤 Decision 和 Action

### 9.2 授权

| 操作 | admin | supervisor | operator |
|------|-------|------------|----------|
| 查看所有数据 | ✓ | ✓ | ✓ (自己的数据) |
| 写入日报 | ✓ | ✓ | ✓ |
| 审批 Decision | ✓ | ✓ | ✗ |
| 修改标准工时 | ✓ | ✗ | ✗ |
| 系统配置 | ✓ | ✗ | ✗ |
| IM 配对审批 | ✓ | ✓ | ✗ |

### 9.3 Agent 安全（新增）

见 [ActionExecutor + IM 集成设计](./2026-06-26-ActionExecutor-IM-设计.md) 第四节：三级风险分级。

- **L0**（只读）：自动执行
- **L1**（低风险写）：语义确认后执行
- **L2**（高风险写）：生成 Decision 卡片，必须 supervisor+ 审批
