## 1. 背景与目标修正

### 1.1 老板真实需求（来自现场录音）

1. **"我实时的能够看到我这台设备用了多少工时"** → 设备维度的**实际工时**统计
2. **"我这人干了多少"** → 人员维度的**实际工时**统计

### 1.2 核心目标修正

| 原方案（v1.0） | 修正后（v2.0） |
|---------------|---------------|
| 交付物：Excel 参数化标准工时计算器 | 交付物：**网页看板系统**（Streamlit） |
| 重点：标准工时预估（新设备报价） | 重点：**实际工时采集 + 标准工时对比** |
| 数据采集：无（假设无法采集） | 数据采集：**两个组长每日报工** |
| 标准工时是唯一产出 | 标准工时是**基准层**，实际工时是**采集层**，对比分析是**展示层** |

### 1.3 为什么修正

- 之前推过多次系统，都是**半成品**，工人用的不爽就 abandon
- 组长（电工+钳工，共15人左右）可以配合每日报工
- 工厂有网络，有 IN3 ERP 系统（可提供账号导出工单）
- 老板要的是**可见性**（看到花了多少），不是**预测性**（估算该花多少）
- 标准工时仍然保留，用于**计划排产**和**效率分析**

## 2. 核心约束

| 约束 | 现状 | 影响 |
|------|------|------|
| 时间 | 两周 | 必须极简可落地 |
| 人力 | 用户（IE）1人 + 两个组长报工 | 用户负责系统搭建+培训，组长负责每日报工 |
| 系统 | 无MES、无扫码 | 无法自动采集，依赖人工报工 |
| 工人 | 不填表 | 报工界面必须面向组长，不能要求工人操作 |
| 网络 | 现场有网络 | 支持网页报工和看板 |
| ERP | IN3，可导出Excel，可能有API | 工单数据来源 |
| 历史教训 | 之前系统都是半成品，强压不持久 | **用户体验决定系统生死** |

## 3. 系统架构：三层分离

```
┌─────────────────────────────────────────────┐
│  老板看板层（Streamlit 桌面端）               │
│  设备工时 | 人员负荷 | 效率偏差 | 计划达成    │
└─────────────────────────────────────────────┘
                      ↑
┌─────────────────────────────────────────────┐
│  数据处理层（Python + Excel 标准工时库）      │
│  关联 ERP 工单 + 组长报工 + 标准工时基准      │
│  算偏差、效率、进度                           │
└─────────────────────────────────────────────┘
                      ↑
        ┌─────────────┼─────────────┐
        ↓             ↓             ↓
   ┌────────┐   ┌──────────┐   ┌──────────┐
   │ IN3 ERP │   │ 组长报工  │   │ 标准工时  │
   │ 工单数据│   │ 移动端页面│   │ 参数化库 │
   └────────┘   └──────────┘   └──────────┘
```

**关键设计决策**：组长报工和老板看板分离
- **组长端**：极简移动端页面，3步完成报工
- **老板端**：Streamlit 桌面看板，图表、对比、趋势

## 4. 数据模型

### 4.1 IN3 ERP 工单数据（数据源）

| 字段 | 说明 | 来源 |
|------|------|------|
| 工单号 | 唯一标识 | IN3 |
| 设备型号 | 项目名 | IN3 |
| 产品族 | A/B/C | IN3 或用户映射 |
| 计划开工日 | | IN3 |

**获取方式**：

### 4.2 组长报工数据（核心采集）

#### 日常简化模式（每天用）

| 字段 | 说明 | 填报人 |
|------|------|--------|
| 日期 | | 系统自动 |
| 员工姓名 | | 组长选 |

| 各工单工时 | 系统默认分配，组长可改 | 组长调 |
| 异常备注 | 可选（等料/5S/返工） | 组长填 |

#### 精确事后补录模式（异常分析时用）

| 字段 | 说明 |
|------|------|
| 日期 | |
| 员工姓名 | |
| 时间段 | 如 8:00-12:00 |
| 并行工单 | 多选 |
| 时长 | 系统自动算 |

**自动切分算法**：
```
A = 4/3 + 2/2 + 6 = 8.33h
B = 4/3 + 2/2     = 2.33h
C = 4/3           = 1.33h
```

### 4.3 标准工时参数化库（基准层）

标准工时是系统的**基准层**，为每个设备/工单提供"应该花多少时间"的参照值，与实际采集的工时对比后产生偏差分析和效率指标。

详细设计见第5章。与系统的结合点：
- **看板计算偏差**：实际工时 - 标准工时 = 偏差金额/时间
- **效率计算**：标准工时 / 实际工时 × 100%
- **计划排产**：新工单用参数化公式预估算工期
- **异常预警**：实际 > 标准 × 1.3 时自动标红
- **数据质量标签**：每个标准工时标记 A/B/C/D/E 可信度等级

## 5. 标准工时参数化详细设计

### 5.1 产品族划分

| 族 | 设备 | 核心结构特征 | 数据来源 |
|---|---|---|---|
| **族A：单机加工设备** | 冲剪机、折弯机、单头铣、双头铣、铜排加工设备 | 床身→立柱/机架→导轨→丝杆传动→夹钳→电机座→调试 | 冲剪机SOP（33步）覆盖约80% |
| **族B：龙门/仓储设备** | 铜排仓、尾料仓 | 龙门立柱→横梁→升降机构→料架→抓手/移栽→仓冲连线 | 无SOP，依赖历史数据+IE估算 |
| **族C：生产线设备** | 20工位抽屉线、打标机产线 | 型材框架→多工位站点→输送轨道/链板线→联动调试 | 无SOP，依赖历史数据+IE估算 |

### 5.3 电工工时参数化

#### 5.3.1 电工工时结构

每台设备电工工时 = 电柜接线工时 + 外围线布线工时

#### 5.3.2 电柜接线工时公式

```
电柜接线工时 = 端子总数 × 单位端子接线时间
             + 断路器/开关数量 × 单件安装基准
             + 继电器/接触器数量 × 单件安装基准
             + 指示灯/按钮数量 × 单件安装基准
             + PLC/控制器数量 × 单件安装基准
             + 控制柜大小系数 × 柜内布线附加时间
```

**参数来源**（需向于立阳/老宋获取）：

| 参数 | 获取来源 |
|------|---------|
| 端子总数 | 设计BOM/接线图 |
| 断路器/继电器/指示灯数量 | 元件清单 |
| 控制柜规格 | 设计图纸（宽×高×深） |
| PLC数量 | 元件清单 |

#### 5.3.3 外围线布线工时公式

```
外围线工时 = 电机功率线布线长度(m) × 单位长度布线基准 × 线径系数
          + 传感器/开关线缆数量 × 单根布线基准
          + 动力柜到设备主线缆长度(m) × 单位长度基准
          + 线槽安装长度(m) × 单位长度线槽安装基准
          + 接线端子压接数量 × 单只压接基准
```

**参数来源**：

| 参数 | 获取来源 |
|------|---------|
| 电机数量及功率 | 设备参数表 |
| 传感器/开关数量 | 设备参数表 |
| 设备到配电柜距离 | 现场布局 |
| 线槽走向长度 | 现场布局 |
| 线缆穿管/桥架 | 现场条件 |

### 5.4 核心修正系数体系

#### 5.4.1 空间可达性系数

| 场景 | 系数 |
|------|------|
| 敞开操作（正面站立） | 1.0 |
| 半受限（需侧身或弯腰） | 1.1~1.2 |
| 受限（需蹲下或仰面） | 1.2~1.4 |
| 盲操作/柜内操作 | 1.4~1.6 |
| 登高操作（需爬梯或平台） | 1.5~2.0 |

#### 5.4.2 吊装/辅助系数（基于现场观察）

| 场景 | 系数 |
|------|------|
| 单人可搬动（<20kg） | 1.0 |
| 双人搬运（20~50kg） | 1.2 |
| 需行吊/叉车（50~200kg） | 1.4~1.6 |
| 需多人+行吊（>200kg） | 1.6~2.0 |
| 大件频繁搬运疲劳附加 | +0.1~0.3 |

#### 5.4.3 对准精度系数

| 精度要求 | 系数 |
|----------|------|
| 目视对准 | 1.0 |
| 螺孔对齐 | 1.1~1.2 |
| 导轨平行度/直线度 | 1.2~1.4 |
| 丝杆同心度/回丝滑入 | 1.3~1.5 |
| 夹钳对中精度 | 1.2~1.4 |

#### 5.4.4 重复操作衰减系数

```
第1颗螺丝 = 1.0
第2~5颗 = 0.95
第6~10颗 = 0.90
第11~20颗 = 0.85
第21颗以后 = 0.80
```

#### 5.4.5 设备型号基础系数

| 设备规模 | 系数 | 典型设备 |
|----------|------|----------|
| 小型 | 0.7~0.8 | 打标机 |
| 中小型 | 0.9~1.0 | 折弯机、单头铣 |
| 中型 | 1.0~1.1 | 双头铣、冲剪机 |
| 中大型 | 1.2~1.3 | 铜排仓 |
| 大型 | 1.4~1.6 | 20工位抽屉线 |

### 5.5 标准工时计算流程

```
输入：设备型号 + 关键参数（导轨长、丝杆数、夹钳数等）
  ↓
匹配产品族（A/B/C）
  ↓
计算各装配单元时间 = 基准时间 × 修正系数1 × 修正系数2 × ...
  ↓
钳工总工时 = Σ(各装配单元时间) × 设备型号系数
  ↓
电工总工时 = 电柜接线 + 外围线布线
  ↓
设备总标准工时 = 钳工总工时 + 电工总工时
  ↓
数据质量标记 + 置信度标注
```

### 5.6 数据质量分级

| 等级 | 含义 | 来源 |
|------|------|------|
| A | 实测 | IE现场秒表测时 |
| B | 历史反推 | 从实际工时拆解得出 |
| C | IE估算 | 基于SOP或经验估算 |
| D | 默认值 | 行业参考值 |
| E | 待验证 | 尚未确认 |

看板中所有标准工时数据都显示等级标签，老板知道哪些可信、哪些仅供参考。

### 5.7 标准工时迭代修正机制（越用越准）

标准工时不是一次算准的，而是**在实际使用中持续校准**的。初始标准工时（数据质量C或D）经过多轮实际采集→偏差分析→基准修正，逐步收敛到真实水平（数据质量升级到B甚至A）。

#### 5.7.1 修正触发条件

系统每周末自动生成修正建议报告，当某个装配单元满足以下条件时触发：

| 条件 | 说明 |
|------|------|
| 样本数 ≥ 3 | 同一装配单元，已有3台以上设备的实际工时数据 |
| 系统性偏差 | 连续多台设备实际工时 vs 标准工时的偏差方向一致（都偏高或都偏低） |
| 偏差幅度 > 15% | 平均偏差超过15%，说明基准时间明显偏离现实 |

**不触发修正的情况**：
- 偏差方向不一致（有的高有的低）→ 说明是现场波动，不是基准问题
- 已知特殊原因（来料不齐、设计变更、新手操作）→ 这些偏差不应用来修正标准

#### 5.7.2 修正流程

```
系统生成修正建议
  ↓
显示：装配单元A-02（导轨安装）
      当前基准：2.5 h/m
      历史实际平均：3.1 h/m
      偏差：+24%（持续偏高）
      建议修正为：3.0 h/m
      样本数：5台
  ↓
IE审核（你）
  ↓
确认修正 → 更新标准工时库 + 记录修正日志
  ↓
数据质量升级：C → B
```

#### 5.7.3 修正规则

| 规则 | 说明 |
|------|------|
| 人工确认 | 系统只给建议，**必须IE点击"采纳"后才生效**，防止自动乱改 |
| 幅度限制 | 单次修正不超过±20%，超过需要分两次修正 |
| 版本记录 | 每次修正记录：旧值→新值、修正日期、修正原因、样本数 |
| 回滚能力 | 修正后效果不佳，可一键回滚到上一版本 |

#### 5.7.4 数据质量升级路径

```
初始：IE估算/历史反推 → 质量C
  ↓
积累3台实际数据 + 修正后 → 质量B
  ↓
IE现场实测验证后 → 质量A
```

#### 5.7.5 在看板中的体现

**新增页面：标准工时健康度**

| 装配单元 | 当前基准 | 历史实际均值 | 偏差 | 样本数 | 状态 | 操作 |
|----------|---------|-------------|------|--------|------|------|
| A-02 导轨安装 | 2.5 h/m | 3.1 h/m | +24% | 5 | 🔴 建议修正 | [采纳] |
| A-05 电机座固定 | 1.8 h/个 | 1.9 h/个 | +6% | 3 | 🟢 正常 | — |

## 6. 组长报工界面设计

### 6.1 核心原则

- **30秒完成**：打开 → 选员工 → 选工单 → 确认数字 → 提交
- **零学习成本**：像填Excel一样直观
- **即时反馈**：提交后显示"今日已报X条，累计Y小时"
- **错误可改**：提交后当天可修改

### 6.2 日常简化界面

```
今日报工 — 2026-05-19

员工：[ 员工甲 ▼]
今日总工时：[ 12 ] h

涉及工单（从IN3拉取）：
☑ 工单WO-240519-01 — 冲剪机YK-800
☑ 工单WO-240519-02 — 铜排仓A型
☐ 工单WO-240519-03 — 抽屉线B型

快捷分配：[ 平均分配 ] [ 主任务优先 ] [ 清空 ]

工单                    工时(h)     占比
─────────────────────────────────────────
冲剪机YK-800 (主)        [ 7.2 ]     60%
铜排仓A型                [ 2.4 ]     20%
抽屉线B型                [ 2.4 ]     20%
─────────────────────────────────────────
合计                     12.0h       ✅

异常备注（可选）：[                    ]

           [ 提交今日报工 ]
```

**交互说明**：
- 点"平均分配"：一键三等分（4/4/4）
- 点"主任务优先"：一键恢复默认（7.2/2.4/2.4）
- 组长可直接修改工时数字，系统实时更新合计
- 合计 ≠ 总工时时标红提示

## 7. 工时切分算法

### 7.1 简化模式算法

**输入**：
- 总工时 T
- 工单列表 W = [w1, w2, ..., wn]
- 主任务 w_main（可选）
- 分配策略 S ∈ {平均, 主任务优先, 自定义}

**输出**：各工单工时 H = [h1, h2, ..., hn]

**规则**：

| 策略 | 计算方式 |
|------|---------|
| 平均分配 | hi = T / n |
| 主任务优先（默认） | h_main = T × 60%，其余 hi = (T × 40%) / (n-1) |
| 自定义 | hi = 组长输入的数字，约束 Σhi ≈ T（允许±0.5h误差） |

### 7.2 精确模式算法

**输入**：
- 时间段列表 P = [(t1_start, t1_end, W1), (t2_start, t2_end, W2), ...]
- 其中 Wi 是该时间段的并行工单集合

**输出**：各工单累计工时

**规则**：
```
for each 时间段 (start, end, W):
    duration = end - start
    for each 工单 w in W:
        H[w] += duration / |W|
```

**示例**：
```
8:00-12:00  A+B+C  4h → A+=4/3, B+=4/3, C+=4/3
12:00-14:00 A+B    2h → A+=2/2, B+=2/2
14:00-20:00 A      6h → A+=6
结果：A=8.33, B=2.33, C=1.33
```

## 7. 老板看板（Streamlit）

### 7.1 页面结构

**左侧边栏**：筛选器
- 日期范围
- 产品族（A/B/C）
- 设备型号
- 组长/人员

**主页面**：

#### 页面1：今日总览
- 在制工单数
- 今日总报工工时
- 人均工时
- 在岗人数

#### 页面2：设备维度
- 每台设备累计实际工时 vs 标准工时
- 偏差率 = (实际 - 标准) / 标准
- 偏差率 > ±30% 标红
- 趋势图（近7天/近30天）

#### 页面3：人员维度
- 每人今日/本周/本月工时
- 人员负荷热力图
- 并行项目数分布

#### 页面4：工单进度
- 甘特图式进度条
- 计划工期 vs 实际已耗工时
- 预计完工时间（基于标准工时和当前效率）

#### 页面5：效率看板
- 工时效率 = 标准工时 / 实际工时 × 100%
- 设备效率排名
- 人员效率排名
- 异常工时占比（等料/5S/返工）

### 7.2 技术栈

- **前端**：Streamlit（Python）
- **数据存储**：Excel 文件（标准工时库 + 报工记录）
- **图表**：Plotly / Altair（Streamlit 原生支持）
- **部署**：本地运行（`streamlit run app.py`），生成本地URL，同一局域网内可访问

## 8. IN3 ERP 集成方案

### 8.1 三级降级策略

| 优先级 | 方式 | 实现 | 稳定性 | 两周内可行性 |
|--------|------|------|--------|-------------|
| 1 | 抓包直调 API | 分析 IN3 网页 Network 请求，Python `requests` 直接调用后端接口 | 高 | 需测试，1-2天可验证 |
| 2 | Playwright RPA | 无头浏览器自动登录 IN3 → 导出工单 Excel → 后台读取 | 中 | 3-5天开发 |
| 3 | 手工导入 | 每日从 IN3 导出 Excel，上传到系统 | 100% | 立即可用 |

### 8.2 接口抽象设计

无论哪种方式，最终都输出统一格式的 DataFrame：

```python
erp_orders = pd.DataFrame({
    '工单号': ['WO-240519-01', ...],
    '设备型号': ['冲剪机YK-800', ...],
    '产品族': ['A', ...],
    '计划开工日': [...],
    '计划完工日': [...],
    '状态': ['在制', ...]
})
```

下游逻辑（报工界面、看板计算）完全不变，只改数据入口。

## 11. 风险与退路

| 风险 | 触发条件 | 退路 |
|------|---------|------|
