## 标准工时现状总结

### 已完成

- **数据获取**：能拿到的数据基本都拿到了——ERP工单240个、报工记录4152条、BOM展开1602条，历史150天的报工数据完整覆盖
- **算法建立**：`标准工时 = 子族Q1基准 × 关键词系数 × 学习宽放`，数学推导，可解释、可追溯
- **产品族覆盖**：13个子族已识别并建立基准（冲剪机、折弯机、单头铣、双头铣、打标机、铜排仓系列、抽屉线系列、横梁移栽等）
- **置信度体系**：A/B/C/D四级，当前分布——A级28个工单（高置信）、B级12个、C级62个、D级1个

### 当前瓶颈

| 问题 | 说明 |
|------|------|
| **C级占比过高（60%）** | 62个工单是C级（低置信），主因是产品族内样本不足或变异大 |
| **缺少秒表测时** | 当前基准全部从历史报工数据反推，没有IE现场观测数据交叉验证 |
| **品种覆盖有限** | 13个子族是基于现有103个已完成工单识别的，ERP里240个工单还有很多产品族未覆盖 |

### 下一步路径

```
当前：历史报工数据 → 统计推导 → 基准（C级为主）
    ↓
中期：+ 秒表测时/观测记录 → 交叉验证 → 提升到A/B级
    ↓
远期：+ 品种扩展 → 全产品线覆盖 → 自动进化
```

**核心判断**：算法框架已经建立，数据管道已经打通（ERP自动同步），下一步不是改算法，而是**用现场观测数据喂给它**。秒表测时是提升置信度最直接的手段。

## AI-Driven 的反思

### 上周验证了什么

上周做的标准工时计算引擎是**真正的AI-driven**：AI读数据→自动识别产品族→推导统计模型→生成公式。这套方法论是AI分析出来的，不是人设计的。

### 本周退回到了什么

本周做的数据库、ERP同步、新页面——本质上是**传统软件工程**。AI帮写了代码，但系统运行时没有AI在思考。11个页面全部是"读数据库→展示数据"。

### 意识到的架构问题

```
上周：数据 → AI分析 → 产出模型（一次性）
本周：数据 → 存数据库 → 页面展示（传统CRUD）
缺失：数据 → AI持续处理 → 洞察+建议 → 人审核 → 系统学习
```

### 架构重新设计

本周后半段重新设计了系统架构，核心理念变化：

| | 之前的设计 | 新的设计 |
|---|---|---|
| AI的角色 | 分析脚本，跑一次就完 | **系统内核**，数据必经之路 |
| 记忆系统 | 变更日志 | **AI的大脑**：记住人的判断，学习工厂规律 |
| 页面 | 数据展示 | 数据 + **AI标注** + 人机交互 |
| 人的角色 | 看数据自己判断 | **审核AI的建议**，批准/否决 |

已输出架构文档：`docs/specs/2026-05-28-AI-Driven架构演进计划.md`

## 工作单元故事

### 背景

上周厂长晨会确认了"工厂日常管理"的产品定位，提出了7个功能方向。本周的任务是把上周的分析看板变成一个**可运行的系统**。

### 过程

**前半周（5/22-5/25）**：基建冲刺
- 把散落的CSV数据迁入SQLite
- 对接IN3 ERP API，实现工单/缺料/BOM/库存实时同步
- 新建4个页面，原有页面全部切换到新数据源
- 建立记忆表基础结构

**后半周（5/26-5/28）**：反思与重设计
- 复盘发现：系统有数据了，但AI不在系统里
- 重新审视"AI native"的含义——不是"AI帮你写代码"，是"AI在系统里持续工作"
- 设计AI处理层架构（core/ai_engine.py），定义每个模块的AI化路径
- 记忆系统从"日志"重设计为"学习系统"

### 关键认知

这周最大的收获不是代码产出，是一个架构理念的转变：

> **系统的价值不在于"有数据可以看"，而在于"AI能从数据中发现人发现不了的东西"。数据库和ERP同步是必要的基建，但如果停在这里，就是一个普通的MIS系统——和市面上的没什么区别。AI-driven的差异在于：数据进来后，AI先处理，人再审核，系统越用越准。**
