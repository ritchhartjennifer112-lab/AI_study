# 工厂日常管理系统 — AI-Driven 架构演进计划

**日期**: 2026-05-28（v2 重写）
**状态**: 草案，待对齐
**核心理念**: AI不是工具，是系统的核心组件。每个模块的数据流都应该经过AI处理层。

---

## 零、设计理念

### 传统工厂系统 vs AI-Driven 工厂系统

```
传统：  数据 → 存储 → 展示 → 人看 → 人决策
AI-driven：数据 → AI处理 → 洞察+建议 → 人审核 → 系统学习 → AI更强
```

| | 传统 | AI-Driven |
|---|---|---|
| 数据进来后 | 存数据库 | **先过AI处理层**，再存 |
| 页面展示 | 原始数据 | 数据 + **AI标注**（异常？趋势？建议？） |
| 人的角色 | 看数据 → 自己判断 | **审核AI的建议** → 批准/否决 |
| 记忆系统 | 变更日志 | **AI的学习素材**：人的判断修正AI模型 |
| 标准工时 | 跑脚本算一次 | 每次新数据**自动重算** + 置信度变化标注 |
| 系统进化 | 人改代码 | 数据越多 → AI越准 → 人干预越少 |

### 三层核心架构

```
┌───────────────────────────────────────────────────────────────┐
│                     Streamlit 页面层                           │
│  每个页面 = 数据展示 + AI标注 + 人机交互                       │
│  ┌─────────┬─────────┬─────────┬─────────┬─────────┐         │
│  │ 报工    │ 总览    │ 工单    │ 缺料    │ 配置    │  ...     │
│  └────┬────┴────┬────┴────┬────┴────┬────┴────┬────┘         │
│       │         │         │         │         │               │
├───────┼─────────┼─────────┼─────────┼─────────┼───────────────┤
│       ▼         ▼         ▼         ▼         ▼               │
│                  AI 处理层 (core/ai_engine.py)                 │
│  ┌─────────────────────────────────────────────────────┐      │
│  │  异常检测    │ 趋势推断    │ 规则推理    │ 建议生成   │      │
│  └─────────────────────────────────────────────────────┘      │
│       ▲         ▲         ▲         ▲         ▲               │
├───────┼─────────┼─────────┼─────────┼─────────┼───────────────┤
│       │         │         │         │         │               │
│                  数据层 (SQLite factory.db)                    │
│  ┌────────────┬────────────┬────────────┬────────────┐        │
│  │ ERP缓存    │ 现场数据   │ AI记忆     │ 配置       │        │
│  │ (同步)     │ (人工)     │ (自动)     │            │        │
│  └────────────┴────────────┴────────────┴────────────┘        │
└───────────────────────────────────────────────────────────────┘
```

**关键原则**：
1. **AI处理层是数据到页面的必经之路** — 不是可选的分析脚本，是系统内核
2. **人的判断是AI的训练信号** — 每次人审核/否决AI建议，都是在教AI
3. **记忆 = AI的大脑** — 不是日志，是AI积累的工厂知识

## 一、AI 处理层设计

### 1.1 核心模块：core/ai_engine.py

```python
class AIEngine:
    """系统核心：所有数据经过此模块处理后再展示。"""

    def process_daily_report(self, report):
        """报工数据进入 → AI处理 → 返回标注。"""
        # 1. 异常检测：偏差率、效率骤降、人员异常
        # 2. 关联分析：这个工单的进度vs交期风险
        # 3. 建议生成：是否需要预警？是否建议调整？

    def process_erp_sync(self, sync_result):
        """ERP数据同步 → AI处理 → 返回洞察。"""
        # 1. 新工单自动分类（产品族、工种需求）
        # 2. 缺料影响评估（哪些工单会延期）
        # 3. 交期风险重算

    def process_memory(self, event):
        """事件进入 → AI记忆更新 → 返回学习结果。"""
        # 1. 人的判断记录到记忆
        # 2. 检查是否有模式可学习
        # 3. 更新AI模型参数
```

### 1.2 AI能做什么（按模块）

| 模块 | 当前（展示层） | AI处理后 |
|------|---------------|----------|
| **报工** | 填表提交 | 提交时实时计算偏差率，异常自动标红+要求填原因 |
| **工单进度** | 显示进度条 | 基于历史同类工单曲线预测完成时间，交期风险从"剩几天"升级为"按当前进度预计X天完成" |
| **缺料** | 展示清单 | 缺料自动关联受影响工单，评估延期天数，给出优先级建议 |
| **标准工时** | 显示基准值 | 新数据进来自动重算，置信度变化自动标注，趋势可视化 |
| **人员** | 显示工时汇总 | 效率趋势检测（某人效率持续下降→预警），技能匹配建议 |
| **加班** | 显示月度累计 | 模式识别（连续加班→疲劳预警），提前干预建议 |

### 1.3 人的审核闭环

```
AI产出建议 → 页面展示（带AI标注）→ 人审核
                                      ├── 批准 → 记忆：AI判断正确，强化
                                      ├── 否决 → 记忆：AI判断错误，修正
                                      └── 补充 → 记忆：AI缺信息，下次参考
```

**记忆存储**：
- `memory_decisions` — 人对AI建议的每次判断
- `memory_corrections` — AI模型被修正的记录
- `memory_patterns` — AI从数据中发现的规律

## 二、记忆系统重设计

### 2.1 从"日志"到"AI大脑"

当前记忆是日志：`谁改了什么 → 记下来`

AI-driven记忆是学习系统：

| 记忆类型 | 用途 | 例子 |
|----------|------|------|
| **决策记忆** | 记住人的判断 | 厂长说"改制件工时打三折" → AI下次自动套用 |
| **修正记忆** | 记住AI犯过的错 | AI预测工单3天完成，实际用了7天 → 修正预测模型 |
| **模式记忆** | 从数据中发现规律 | "威胜的铜排仓每次都比标准多50%" → 自动调整系数 |
| **上下文记忆** | 记住工厂的特殊情况 | "春节前两周效率普遍下降" → 季节性调整 |

### 2.2 记忆驱动的闭环

```
新数据进来
    ↓
AI查记忆 → "以前类似情况怎么处理的？"
    ↓
AI生成建议（参考历史判断）
    ↓
人审核 → 批准/否决/补充
    ↓
结果写入记忆 → AI下次更准
```

### 2.3 记忆表结构（重设计）

```sql
-- 决策记忆：人对AI建议的判断
CREATE TABLE memory_decisions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    context TEXT NOT NULL,          -- 什么场景下
    ai_suggestion TEXT NOT NULL,    -- AI建议了什么
    human_decision TEXT NOT NULL,   -- 人怎么决定的
    reason TEXT,                    -- 人给的原因
    outcome TEXT,                   -- 最终结果（事后回填）
    created_at TEXT NOT NULL DEFAULT (datetime('now','localtime'))
);

-- 修正记忆：AI模型被修正
CREATE TABLE memory_corrections (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    model_type TEXT NOT NULL,       -- 'std_time' / 'delivery_predict' / 'efficiency'
    entity_id TEXT NOT NULL,        -- 工单号/员工/产品族
    old_prediction TEXT,            -- AI之前的预测
    actual_result TEXT,             -- 实际结果
    correction_reason TEXT,         -- 为什么差这么多
    created_at TEXT NOT NULL DEFAULT (datetime('now','localtime'))
);

-- 模式记忆：AI从数据中发现的规律
CREATE TABLE memory_patterns (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    pattern_type TEXT NOT NULL,     -- 'product_coefficient' / 'seasonal' / 'customer_habit'
    pattern_key TEXT NOT NULL,      -- 如"威胜铜排仓"、"春节前"
    pattern_value TEXT NOT NULL,    -- 如"工时系数1.5"、"效率下降20%"
    confidence REAL DEFAULT 0.5,    -- 模式置信度，0~1
    times_confirmed INTEGER DEFAULT 1,
    last_confirmed_at TEXT,
    created_at TEXT NOT NULL DEFAULT (datetime('now','localtime'))
);

-- 上下文记忆：工厂的特殊情况
CREATE TABLE memory_context (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    category TEXT NOT NULL,         -- 'customer' / 'product' / 'seasonal' / 'policy'
    key TEXT NOT NULL,              -- 客户名/产品族/时间段
    value TEXT NOT NULL,            -- 记住什么
    source TEXT,                    -- 谁说的/哪来的
    created_at TEXT NOT NULL DEFAULT (datetime('now','localtime'))
);
```

## 三、模块级AI设计

### 3.1 标准工时（已验证的AI能力 → 系统化）

**现状**：`_calc_standard_times.py` 跑一次出JSON，页面读静态数据

**AI-driven改造**：
- 报工数据进入时自动触发标准工时重算
- 置信度变化自动标注（"从C级升到B级"）
- 效率异常自动检测（"这个工单效率31%，远低于同类工单"）
- **人可以修正AI的计算参数**（如调整某子族的系数）→ 记入修正记忆

```
新报工 → 自动重算标准工时 → 置信度变化? → 写入memory_corrections
                                         → 效率异常? → 写入memory_events
                                         → 页面标注变化
```

### 3.2 交期风险（从规则到预测）

**现状**：`delivery_risk.py` 算"离交期还剩几天" → 红黄绿

**AI-driven改造**：
- 基于历史同类工单的完成曲线做预测
- 考虑当前在手工时、人员分配、缺料情况
- 不是简单的"剩几天"，是"按当前进度，这个工单大概率X天完成"
- 如果预测超过交期 → 提前预警 + 给出建议（加人？加班？调整优先级？）

```
工单进度 + 历史曲线 + 缺料情况 → 预测完成天数 → vs交期
    → 风险? → AI建议（加人/加班/调整优先级）→ 人审核
    → 每日快照 → 趋势分析 → 风险升级预警
```

### 3.3 缺料跟踪（从展示到影响评估）

**现状**：展示ERP缺料清单

**AI-driven改造**：
- 缺料进来 → 自动关联受影响的工单
- 评估"这25项缺料会导致哪些工单延期、延期几天"
- 给出优先级建议（先催哪个物料影响最大）
- 库存趋势分析（"这个物料连续3天下降，预计下周断货"）

```
ERP缺料同步 → AI关联工单 → 评估延期影响 → 优先级排序
    → AI建议："先催施耐德空开，影响3个工单共15天"
    → 人审核 → 采购行动 → 结果记录到记忆
```

### 3.4 报工系统（从填表到智能辅助）

**现状**：填表 → 存数据库

**AI-driven改造**：
- 提交时实时计算偏差率
- 异常自动检测 + 要求填原因
- 历史参考："上次这个工单类似阶段用了X小时"
- 学习曲线提示："这是该产品第3次生产，预期效率比首次高20%"

### 3.5 人员维度（从统计到洞察）

**现状**：显示工时汇总

**AI-driven改造**：
- 效率趋势检测：某人效率持续下降 → 预警（疲劳？技能瓶颈？）
- 技能匹配：基于历史工单数据，AI推荐"这个钳工做过3次冲剪机，适合派这个活"
- 加班模式识别：连续加班 → 疲劳预警 → 建议调整排班

## 四、实施路线

### Phase 1：AI处理层骨架 + 记忆系统重设计（本周）

**目标**：建立AI作为系统核心组件的架构基础。

**交付物**：
1. `core/ai_engine.py` — AI处理层入口
   - `process_daily_report()` — 报工异常检测（偏差率、效率骤降）
   - `process_erp_sync()` — ERP同步后自动评估影响
   - 每个方法返回结构化的AI标注（anomaly/warning/suggestion）

2. 记忆表重设计
   - 新建 memory_decisions / memory_corrections / memory_patterns / memory_context
   - 迁移现有 memory_events 中的有效数据

3. 报工页面接入AI
   - 提交时调用 ai_engine.process_daily_report()
   - 异常自动标红 + 要求填原因
   - 结果写入记忆

**验证**：报工提交时能看到AI标注，异常有原因记录。

### Phase 2：交期预测 + 缺料影响评估（下周）

**目标**：从"展示数据"升级为"AI评估+建议"。

**交付物**：
1. 交期预测算法
   - 基于历史同类工单曲线预测完成时间
   - 工单进度页显示预测值 vs 交期
   - 风险升级时生成AI建议

2. 缺料影响评估
   - 缺料自动关联工单 + 评估延期天数
   - 优先级排序建议
   - 页面显示"AI评估：此缺料影响3个工单，预计延期15天"

3. 人审核闭环
   - AI建议页面展示 + 批准/否决按钮
   - 判断结果写入 memory_decisions

**验证**：工单页有预测完成时间，缺料页有影响评估，有审核操作记录。

### Phase 3：记忆驱动的学习（第3周）

**目标**：系统开始从人的判断中学习。

**交付物**：
1. 模式发现
   - 从效率数据中自动发现客户/产品/季节性规律
   - 写入 memory_patterns，带置信度

2. 记忆查询
   - AI生成建议时自动查询历史相似场景
   - "上次厂长说改制件怎么处理的？" → 从记忆中检索

3. 标准工时自动进化
   - 新数据进来自动重算 + 置信度变化标注
   - 人修正参数时写入 memory_corrections
   - AI参考修正历史优化预测

**验证**：AI建议引用历史判断，标准工时置信度有变化趋势，模式记忆有积累。

### Phase 4：排产建议（远期，数据积累后）

**目标**：AI基于约束给出排产方案。

- 输入：待排工单 + 可用资源 + 记忆中的规律
- 约束：交期、工种、工序依赖、资源容量
- 输出：AI排产方案 + 理由
- 人审核调整 → 记入记忆 → AI下次更准

**这个阶段需要Phase 1-3的数据积累。**

## 五、与上一版架构计划的关系

上一版（2026-05-27）描述的是**数据基础设施**——数据库、ERP同步、表结构。

本版（2026-05-28）描述的是**AI作为核心组件**的系统设计。

两版是**叠加关系**，不是替代：

| | v1（数据基建） | v2（AI-driven） |
|---|---|---|
| 关注点 | 数据怎么存、怎么同步 | 数据怎么处理、怎么用 |
| 核心模块 | database.py, erp_sync.py | ai_engine.py |
| 记忆定位 | 变更日志 | AI的学习系统 |
| 页面定位 | 数据展示 | 数据 + AI标注 + 人机交互 |
| 人的角色 | 看数据做决策 | 审核AI建议 |

**v1已完成的部分**（Phase 1 ERP同步 + 数据库 + 记忆表基础结构）直接作为v2的数据层，不需要重做。

**需要新增的**：AI处理层（core/ai_engine.py）+ 记忆系统升级 + 页面接入AI标注。

## 六、关键设计决策

| # | 决策 | 原因 |
|---|------|------|
| 1 | **AI处理层是必经之路，不是可选插件** | 如果AI是可选的，人就会跳过它，系统退化为传统工具 |
| 2 | **人的审核是AI的训练信号** | 每次批准/否决都在教AI，这是系统越用越聪明的关键 |
| 3 | **记忆不是日志，是AI的大脑** | 日志记录"发生了什么"，记忆记录"学到什么" |
| 4 | **AI建议必须可解释** | 不能是黑盒。每个建议都要附带"基于什么数据/规律得出" |
| 5 | **渐进式AI介入** | 初期AI只做异常检测（低风险），积累信任后再做预测和建议 |
| 6 | **本地SQLite保留** | 数据量在万级，SQLite够用；单文件部署简单；AI处理在Python层做 |
